# Recommendation for Production Deployment

For production, keep the single-machine architecture only as a development stage and move to an epoch-based indexing model with immutable search snapshots. This keeps query behavior consistent while indexing runs continuously: indexers build mutable segments, then publish atomically to a new epoch; search workers pin one epoch for the full request lifecycle. Pair this with stricter frontier controls (per-domain rate limits, global queue high-watermarks, adaptive worker throttling) so back pressure is proactive instead of reactive. At this stage, persist crawl frontier, visited URLs, and segment manifests to durable storage so restarts do not lose progress.

In deployment, split services by responsibility: crawler/indexer workers, query API, and observability plane. Use PostgreSQL (or similar) for metadata/state, object storage for segment files/snapshots, and a message queue for resilient crawl tasks. Add robust operational controls: robots.txt policy, retry classification, circuit breakers, p95 latency SLOs, and dashboard alerts for queue saturation and indexing lag. This progression preserves the educational design choices of this assignment (native crawling/parsing, explicit concurrency control) while making the system maintainable under sustained load.
